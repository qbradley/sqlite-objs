## Summary

Aragorn successfully implemented a three-binary benchmark harness for performance comparison between vanilla SQLite and sqlite-objs (Azure blob-backed). The solution involved:

1. **Main harness (`benchmark.c`)** — Lightweight orchestrator using `system()` to shell out to speedtest1 subprocesses, measuring wall-clock time via `gettimeofday()`, and producing formatted output (text or CSV)
2. **Standard speedtest1 binary** — SQLite's official speedtest1 benchmark (unmodified upstream source)
3. **Azure-enabled speedtest1 binary** — Same speedtest1 with sqlite-objs VFS registered as default
4. **Build system (`Makefile`)** — Supports both stub (local-only) and production (Azure-enabled) builds
5. **Documentation (`README.md`)** — User guide for running benchmarks with various options

## Design Decisions

**Chosen Approach:** Three-binary subprocess architecture

**Why not embed speedtest1 directly?**
- speedtest1.c calls `exit()` directly from main, terminating the process before the harness could capture results
- Attempted #define main trick failed for the same reason
- Would require maintaining a fork of upstream speedtest1

**Benefits of subprocess approach:**
- Clean separation of concerns
- Unmodified speedtest1.c from upstream
- Simple, accurate timing measurement
- Flexible output formats (text, CSV)
- Supports both development (stub Azure) and real Azure benchmarks

**Tradeoffs accepted:**
- ~10ms subprocess overhead per invocation (negligible for multi-second benchmarks)
- Only captures total elapsed time, not per-test breakdowns (users can run binaries directly for details)
- No shared memory between harness and subprocess

## Implementation

### File Structure
```
benchmark/
├── benchmark.c           — Main harness
├── speedtest1.c          — Official SQLite speedtest1 (unmodified)
├── speedtest1_wrapper.c  — VFS registration wrapper for Azure build
├── Makefile              — Build system
├── README.md             — User documentation
├── build/                — Build artifacts directory
```

### Command-Line Options
- `--local-only` — Run only local SQLite benchmark
- `--azure-only` — Run only sqlite-objs benchmark
- `--size N` — Database size factor (default 10)
- `--output csv|text` — Output format (default: text)

### Example Usage
```bash
# Build stub (local-only)
cd benchmark && make

# Run local benchmark
./benchmark --local-only --size 25

# Build with Azure support
make all-production

# Run full comparison with Azure env vars
export AZURE_STORAGE_ACCOUNT=myaccount
export AZURE_STORAGE_KEY=mykey
export AZURE_STORAGE_CONTAINER=benchmarks
./benchmark --size 50

# CSV output for automation
./benchmark --output csv > results.csv
```

## Exit Code Handling

speedtest1 returns 1 if optional features (e.g., rtree) are missing. The harness treats both 0 and 1 as success to avoid false failures. Only non-zero exit codes indicate actual errors.
